Quick Start

There are two main ways to install it. You can try your luck with our install script or follow the steps. But before anything, you need to meet the requirements.

Requirements: JDK >= 1.6

Install script

This script will try to resolve all dependencies (except JDK) and get your cube up and running.

$ curl -s http://zalando.github.io/cube/install.sh | /bin/bash -s --
Manual install

You can also try to do it manually by following these steps:

  1. Solve dependencies

    • Maven 2.2
    • ImageMagick
    • Git
    • Nodejs >= 0.8
    • npm

  2. Get the cube from our GitHub repo

    $ git clone git@github.com:zalando/cube.git

  3. Install node dependencies

    $ cd cube
    $ npm install .

  4. Build and Run Solr Database

    $ mvn clean install tomcat:run-war -Pdevelopment

  5. Start the Cube on a new terminal

    $ cd cube
    $ coffee cube.coffee

  6. Visit http://localhost:3000

Create your own Cube: Library example.

Once you got the included example up and running, it will be no hassle to set up the cube to handle your own data. In the following small tutorial, I'll guide you through the process of creating a Library inside the cube. Adding your own data would be a matter of writing a different schema and small configuration changes. Let's get started!

  1. Add a solr core

    Edit your solr/solr/solr.xml file. At the end it should look something like this

    <?xml version="1.0" encoding="UTF-8" ?>
<solr sharedLib="lib" persistent="true">
  <cores adminPath="/admin/cores">
    <core name="team" instanceDir="conf/cube/" dataDir="conf/cube/team" />
    <core name="library" instanceDir="conf/cube/" dataDir="conf/cube/library" />
  </cores>
</solr>
  2. Start Solr

    * stop it with Ctrl+C if its running already

    $ cd solr/
    $ mvn clean install tomcat:run-war -Pdevelopment

    You should have a new core called "library" by now. Check it on http://localhost:38730/cube-solr. This is were all our library collection will be stored.

  3. Add it to entities.json

    The library collection and the team collection are called 'entities' for the cube. An 'Entity' is a collection of items that share the same properties, or schema. By now we have 2 entities in the Cube: team and library, but we need to add it to the entities.json file so Cube knows about it. Please weatch out that it is a json file! be careful with the syntax. At the end it should look like this:

    [ "team", "library" ]
  4. Create entity default files
    $ cd util/
    $ ./create_entity library
  5. Edit to suit your needs

    1. Schema

      The Schema is one of the most important aspects of your entity. We recommend you to spend some time checking out the one provided in the example along with its documentation.

      Please check the Schema reference here.

      For this example, I provide you with a schema. Get it from here and place it in entities/library/

      $ cp schema.json cube/entities/library/

    2. Database config

      Configrue entities/library/db.json to point to the new Solr core

      It should look something like this:

      {
    "host": "localhost",
    "port": "38730",
    "core": "library",
    "path": "/cube-solr"
}

    3. Entity Information

      We need to provide some information about the entity along with some settings. Please take a look at entities/library/settings.json. It should look something like this:

       {
    "entity": "library",
    "title": "Library",
    "description": "My huge book collection",
    "itemType": [ "book", "books" ],
    "separator": "/",
    "view": "list",
    "rows": 50,
    "sort": "title:asc"
}
  6. Start NodeJs
    $ coffee cube.coffee
  7. Visit your library
    http://localhost:3000/library/?admin=yes
  8. Add your books!

    Click on the top right corner and start adding your books!

Schema reference

A Cube Schema is a json array of objects which describe the data structure of an entity. You can take a look at an example schema which makes uses of almost all possible properties. Setting up yours is a matter of listing the fields with their IDs and Labels (minimum required properties) and adding/removing/negating some properties on each of them.

id:

Each field must have a unique ID composed of alphanumeric characters. This field is required.

Value: String

label:

IDs are not necessarily pretty or even human readable. A label is a must. Alphanumeric characters and spaces.

Value: String

type:

The type field specifies which data type this field contains. It tells the cube how to store this value and also how to display it. Default (no type property specified) is String. Other data types are facet, img, email, multiline, etc. Choose one from the list below, but don't forget to quote it.

  • integer

    This field will store integer numbers, not strings. Useful when you want to sort by number value instead of string value

  • float

    This field will store float numers, not strings. Useful when you want to sort by number value with decimals instead of string value. By default is set to round to 2 decimal digits.

  • facet

    This field will act as a facet field. It will appear on the facet index, listing all values that your items have in this field, along with a count of items with each value.

  • tuple

    A tuple field allows you to relate 2 facet fields to each other. Lets say you have items that have a group property and a role property, and that one item can have a role related to one group, but another role in a different group. Then you would use a tuple field to keep this information.

  • img

    Image field. Items can have only one image field. This will be displayed as the profile picture on the profile view, it will activate the thumbnail view and it will be displayed as a small 25x25 pic on the list view.

  • email

    Stores emails and renders them as clickable emailto: links.

  • skype

    Stores skype usernames and renders them as clickable skypeto: links to open a chat window.

  • multiline

    Multiple line field, renders a textarea instead of an input field.

  • dropdown

    Shows you a <select> element, showing you a list of selectable options. This field will require you to define the options in an array under the property "options".

  • date

    Shows a date and attaches a date picker for input.

Value: String (facet | tuple | img | email | skype | multiline | dropdown | date)

mandatory:

This field will be required for form validation when adding/editing new items

Value: Boolean (true | false)

search:

This field is a searchable field. This means that when you type something in the search input field, it will look in this and any other searchable field.

Value: Boolean (true | false)

index:

Show this field in the list view as a column.

Value: Boolean (true | false)

thumbnail:

Display this field as a footer for the item views in Thumbnail mode.

Value: Boolean (true | false)

multiedit:

This field is editable on many items at once. It will appear on the multiedit form

Value: Boolean (true | false)

hidden:

When defined true, hides this field from the profile view

Value: Boolean (true | false)

additional:

This field will be part of the additional fields and will be showed at the bottom of the profile view, hidden inside the 'Additional info' section

Value: Boolean (true | false)

Type specific properties

Facet
specials:

Each facet is splitted in two areas, separated by a gray line. Define this property with an array of strings and each facet field that matches will be displayed below the line.

Value: Array [ "name1", "name2" ]

classifier:

Defines this field as the category to sort items on thumbnail view. The order of this array will determine the order of the sections on that view.

Value: Array [ "name1", "name2" ]

Dropdown
options:

Required property for any dropdown tipe field, defined as an array of strings for the availablo options in the dropdown menu.

Value: Array [ "name1", "name2" ]
